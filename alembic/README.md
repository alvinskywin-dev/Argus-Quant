# Database migrations (Alembic)

Versioned schema for ARGUS QUANT. The DSN and target metadata come from the app
(`app.config.settings.postgres_dsn`, `app.database.models.Base.metadata`), so
migrations always match the running code — no credentials live in the repo.

## Layout
- `alembic.ini` (repo root) — config; URL injected by `env.py`.
- `alembic/env.py` — async (asyncpg) migration runner; imports all model modules.
- `alembic/versions/0001_baseline.py` — full current schema (adoption baseline).

## Commands (run from the repo root)

```bash
# Brand-new database → build the full schema and record the version:
alembic upgrade head

# Existing database already provisioned by the legacy on-boot DDL →
# mark it at the baseline WITHOUT running any DDL:
alembic stamp head

# Inspect state:
alembic current
alembic history

# Generate SQL without touching a DB (review before applying):
alembic upgrade head --sql

# Author a new change after editing models:
alembic revision --autogenerate -m "describe change"
```

Inside the bot container: `docker compose exec bot alembic upgrade head`.

## Adoption / transition notes
- The legacy `init_db()` path (`Base.metadata.create_all` + idempotent
  `_SCHEMA_UPGRADES`) is still in place as a non-destructive fallback. New
  environments should switch to `alembic upgrade head`; existing ones should be
  `alembic stamp head` once, then managed by Alembic going forward.
- Autogenerate requires a reachable database; the baseline here is built from the
  ORM metadata (not autogenerated) so it is DB-independent and safe to adopt
  without a live connection. New incremental revisions can be autogenerated
  against a dev/staging DB.
- Keep new revisions small and reversible; prefer `op.*` operations over raw SQL
  so they work across environments.
